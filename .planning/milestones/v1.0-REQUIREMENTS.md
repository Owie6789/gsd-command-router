## Phase 1: Foundation & Search

### SEARCH-01: Runtime Filtering

**As a** user of a specific AI tool (e.g., Claude Code)  
**I want** to filter commands by runtime  
**So that** I only see commands compatible with my setup

**Acceptance Criteria:**
- [ ] Filter chips show all 15 supported runtimes (Claude Code, GPT Engineer, Gemini, etc.)
- [ ] Clicking a runtime chip filters command list to show only commands supporting that runtime
- [ ] Runtime filter works independently of category filter
- [ ] Active runtime filter is visually indicated (highlighted chip)
- [ ] Clicking active runtime chip again removes the filter

**Notes:**
- Runtimes sourced from existing `RUNTIMES` array in `gsd.html`
- Each command has `runtimes` property listing supported tools
- Filter logic: `command.runtimes.includes(selectedRuntime)`

---

### SEARCH-02: Combined Filtering

**As a** power user  
**I want** to combine category + runtime filters  
**So that** I can narrow down to exactly what I need

**Acceptance Criteria:**
- [ ] User can select one category AND one runtime simultaneously
- [ ] Combined filter shows commands matching BOTH criteria (AND logic)
- [ ] UI clearly shows both active filters
- [ ] Either filter can be cleared independently
- [ ] "Clear all" button resets both filters at once

**Notes:**
- Current code has `activeFilter` for category — extend to support runtime
- Consider object state: `{ category: 'core', runtime: 'Claude Code' }`

---

### SEARCH-03: Improved Search Relevance

**As a** user searching for commands  
**I want** more relevant results when I type  
**So that** I find what I need faster

**Acceptance Criteria:**
- [ ] Search matches command name (`/gsd-*`) with higher priority than description
- [ ] Search is case-insensitive
- [ ] Search matches partial words (e.g., "phase" matches "phases")
- [ ] Results are sorted by relevance (name matches first, then description)
- [ ] Empty state shows helpful message when no matches found

**Notes:**
- Current search uses simple `includes()` on all fields
- Implement weighted scoring: name match = 2 points, description = 1 point
- Consider fuzzy matching library (e.g., Fuse.js) if simple approach insufficient

---

### KEYBOARD-01: Arrow Key Navigation

**As a** keyboard-focused user  
**I want** to navigate the command list with arrow keys  
**So that** I don't need to reach for the mouse

**Acceptance Criteria:**
- [ ] Down arrow moves focus to next command row
- [ ] Up arrow moves focus to previous command row
- [ ] Focus wraps around (down from last goes to first, up from first goes to last)
- [ ] Focused command row has visible focus indicator (border or background highlight)
- [ ] Arrow keys work only when search input is NOT focused

**Notes:**
- Track `focusedIndex` in state
- Use `element.focus()` on command rows
- Ensure rows have `tabindex="-1"` for programmatic focus

---

### KEYBOARD-02: Enter/Space Activation

**As a** keyboard-focused user  
**I want** to expand/collapse commands with Enter or Space  
**So that** I can view details without mouse

**Acceptance Criteria:**
- [ ] Pressing Enter on focused command row toggles its expanded state
- [ ] Pressing Space on focused command row toggles its expanded state (prevents page scroll)
- [ ] Escape key closes expanded command (if one is open)
- [ ] Second Escape press clears search (if search has value)
- [ ] Keyboard behavior matches mouse click behavior exactly

**Notes:**
- Current code has `openCmd` state for expanded command
- Ensure `e.preventDefault()` on Space to prevent page scroll
- Update `aria-expanded` attribute for screen readers

## Phase 2: State & Persistence

### STATE-01: View Persistence

**As a** user navigating between views  
**I want** my last active view to persist across page reloads  
**So that** I return to where I was

**Acceptance Criteria:**
- [ ] Active view (`activeView` state) is saved to localStorage on every view switch
- [ ] On page load, app restores last active view from localStorage
- [ ] If no saved view exists (first visit), default to "commands" view
- [ ] View restoration happens before first render (no flash of wrong view)

**Notes:**
- Use `localStorage.setItem('gsd-activeView', viewId)`
- Restore in initialization code before `renderCommands()`

---

### STATE-02: Filter Persistence

**As a** user who sets up filters  
**I want** my search query and filters to persist across reloads  
**So that** I don't lose my context

**Acceptance Criteria:**
- [ ] Search query (`searchVal`) is saved to localStorage on every change
- [ ] Active category filter is saved to localStorage on every change
- [ ] Active runtime filter is saved to localStorage on every change
- [ ] On page load, all filters are restored from localStorage
- [ ] "Clear all" button also clears saved state from localStorage

**Notes:**
- Debounce saves to avoid excessive localStorage writes
- Consider single state object: `localStorage.setItem('gsd-filters', JSON.stringify({...}))`

---

### STATE-03: Theme Persistence

**As a** user who selects a theme  
**I want** my theme preference to persist across sessions  
**So that** I don't have to re-select it every time

**Acceptance Criteria:**
- [ ] Theme preference is saved to localStorage when changed
- [ ] On page load, theme is restored before first paint (no flash of wrong theme)
- [ ] Default theme is dark (if no preference saved)
- [ ] Theme toggle button shows current theme state

**Notes:**
- Store as `'gsd-theme': 'dark' | 'light'`
- Apply theme class to `<body>` or `<html>` element
- Use CSS custom properties for theme colors

---

### HISTORY-01: Recently Viewed Commands

**As a** user who browses commands  
**I want** to see my recently viewed commands  
**So that** I can quickly return to commands I was just using

**Acceptance Criteria:**
- [ ] When user expands a command detail, it's added to "recent" list
- [ ] Recent list shows last 10 viewed commands (FIFO)
- [ ] Recent commands are displayed in sidebar or dedicated view
- [ ] Clicking a recent command navigates to it and expands detail
- [ ] Recent list persists in localStorage across sessions
- [ ] User can clear recent history

**Notes:**
- Store as array of command IDs: `['cmd1', 'cmd2', ...]`
- Most recent first (unshift to front, pop from end)
- Avoid duplicates (remove if exists, then unshift)

### FAVORITES-01: Mark Favorites

**As a** power user  
**I want** to mark commands as favorites  
**So that** I can quickly access my most-used commands

**Acceptance Criteria:**
- [ ] Each command row has a star/favorite icon
- [ ] Clicking star toggles favorite status
- [ ] Favorited commands have filled star icon
- [ ] Non-favorited commands have outline star icon
- [ ] Favorite status persists in localStorage

### FAVORITES-02: Filter Favorites

**As a** user with favorited commands  
**I want** to filter to show only my favorites  
**So that** I can see my curated list

**Acceptance Criteria:**
- [ ] "Favorites" filter chip or toggle exists
- [ ] Activating favorites filter shows only favorited commands

## Phase 3: Export & Sharing

### EXPORT-01: Markdown Export

**As a** user who wants offline access  
**I want** to export commands as a Markdown file  
**So that** I can read them in my editor or print them

**Acceptance Criteria:**
- [ ] Export button exists in UI (topbar or sidebar)
- [ ] Export includes currently filtered commands (respects active filters)
- [ ] Markdown format includes: command name, description, flags, usage, when, related
- [ ] File is downloaded as `gsd-commands.md` (or `gsd-commands-{filter}.md`)
- [ ] Export completes within 2 seconds for all 50+ commands

**Notes:**
- Generate Markdown string, create Blob, trigger download
- Format example:
  ```markdown
  ## /gsd-phase

  **Category:** Core  
  **Description:** Create a new phase...
  
  **Flags:**
  - `--name` - Phase name
  
  **Usage:**
  ```
  /gsd-phase --name "Phase 1"
  ```
  
  **When:** When you want to structure work...
  ```

## Phase 5: Performance & Polish

### PERF-01: Search Debouncing

**As a** user typing in search  
**I want** the app to wait until I finish typing  
**So that** it doesn't re-render on every keystroke

**Acceptance Criteria:**
- [ ] Search input has debouncing (300ms delay)
- [ ] Filter only triggers after user stops typing for 300ms
- [ ] Debounce delay feels responsive (not too slow, not too fast)
- [ ] Clearing search triggers immediately (no delay)
- [ ] Debounce is cancelled when user switches views

**Notes:**
- Use `setTimeout` + `clearTimeout` or lodash `debounce`
- Show loading indicator if debounce delay is noticeable

---

### PERF-02: Virtual Scrolling Preparation

**As a** user with hundreds of commands  
**I want** the list to remain performant  
**So that** the app doesn't lag

**Acceptance Criteria:**
- [ ] Code structure supports virtual scrolling (windowing)
- [ ] Only visible commands are rendered to DOM
- [ ] Scroll position is maintained while virtualizing
- [ ] Performance is smooth with 100+ commands
- [ ] Fallback to full render if <50 commands (not needed yet)

**Notes:**
- May not be necessary yet (only 50+ commands)
- Prepare architecture for future growth
- Consider library like `react-window` if adopting React

**So that** I understand the UI structure
